# Plan: Script d'organisation JSON — config/ vs data/

## Objectif

Créer un script Python qui scanne un répertoire source, analyse la structure interne de chaque fichier JSON, et les déplace dans deux sous-répertoires :
- `config/` — fichiers de configuration (clés paramètres, settings, constantes)
- `data/` — fichiers de données textuelles (scénaires, contenu, entrées utilisateur)

## Contexte

Le projet génère des fichiers JSON de natures différentes :
- **Configuration** : `kilo.json` (paramètres plugin), schémas JSON, fichiers avec clés `model`, `api_url`, `temperature`
- **Données** : `scenario_output.json` (scénarios VR), `agent-manager.json` (état sessions), fichiers avec clés `etapes`, `titre`, `scenario_id`

## Spécifications

### Entrée
- Répertoire source (défaut : `./`)
- Répertoire destination (défaut : `./organized/`)

### Logique de classification

Un fichier JSON est classé **config** si :
1. Racine est un `dict` contenant au moins 2 de ces clés : `model`, `api_url`, `api_key`, `token`, `temperature`, `max_tokens`, `timeout`, `host`, `port`, `database`, `version`, `settings`, `dependencies`, `plugins`, `rules`
2. OU racine est un `dict` dont toutes les valeurs sont primitives (str/int/float/bool/null) — pas de nested arrays d'objets

Un fichier JSON est classé **data** si :
1. Racine est un `dict` contenant au moins 1 de ces clés : `scenario_id`, `etapes`, `titre`, `content`, `text`, `messages`, `history`, `data`, `results`, `items`, `records`
2. OU racine est un `list` d'objets (ex: array de scénarios)
3. OU contient des valeurs textuelles longues (>100 chars) dans des champs `text`, `content`, `description`, `page_content`

### Sortie
```
organized/
├── config/
│   ├── kilo.json
│   └── settings.json
└── data/
    ├── scenario_output.json
    └── agent-manager.json
```

## Architecture proposée

```
organize_json.py
├── classify_json(path) -> "config" | "data" | "unknown"
│   ├── load_json(path) -> dict|list
│   ├── has_config_keys(obj) -> bool
│   └── has_data_keys(obj) -> bool
├── organize(source_dir, dest_dir, dry_run=False)
│   ├── scan(source_dir) -> list[Path]
│   ├── classify each file
│   └── move/copy to dest_dir/config or dest_dir/data
└── main() — CLI args
```

## Décisions de design

| Décision | Option A | Option B | Recommandation |
|----------|----------|----------|----------------|
| Action | **Copier** (sûr) | Déplacer (destructif) | **Copier** par défaut, flag `--move` pour déplacer |
| Fichiers ambigus | Erreur | Dossier `unknown/` | **`unknown/`** avec log warning |
| Profondeur scan | 1 niveau | Récursif | **Récursif** avec `--depth` |
| Conflit noms | Écraser | Renommer avec suffixe | **Renommer** (`file_1.json`) |
| Logs | print | logging | **print** (script simple) |

## Risques

| Risque | Impact | Mitigation |
|--------|--------|------------|
| Fichier JSON invalide | Crash script | `try/except` + skip avec warning |
| Fichier volumineux (>100MB) | Mémoire | Stream parsing ou skip > seuil |
| Permissions insuffisantes | Échec copie | Vérification avant + message clair |
| Clés communes aux deux catégories | Classification erronée | Priorité aux clés data (plus spécifiques) |
| Chemins Unicode | Crash Windows | Utiliser `pathlib.Path` partout |

## Validation

1. Créer des fichiers JSON de test dans `/tmp/test_json_org/`
2. Lancer `python organize_json.py /tmp/test_json_org --dry-run`
3. Vérifier la classification
4. Lancer sans `--dry-run` et vérifier la structure de sortie
5. Vérifier les cas edge : JSON invalide, fichier vide, fichier volumineux

## Fichiers concernés

- **Nouveau** : `organize_json.py` (racine du projet)
- **Modifié** : aucun
- **Dépendances** : stdlib uniquement (`json`, `pathlib`, `shutil`, `argparse`)

## Questions ouvertes

1. **L'utilisateur veut-il inclure les fichiers `.json` déjà présents dans `.kilo/` ?** → Recommandation : oui, tout scanner
2. **Faut-il gérer les fichiers JSONL (JSON Lines) ?** → Non, hors scope
3. **L'action par défaut est-elle copier ou déplacer ?** → Copier (plus sûr)
